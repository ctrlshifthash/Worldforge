I Built a Full 2D Game Engine Inside a Next.js App in 10 Days

Worldforge: An AI worldbuilding platform where every world you create becomes a playable pixel-art game with combat, quests, NPCs, multiplayer, and a building system

Ten days ago, Worldforge was a concept: what if you could describe a fictional world in a sentence, have AI generate the entire thing — characters, factions, timeline, map — and then walk into it?

Not view it. Not read about it. Actually walk into a 2D pixel-art game world, fight enemies, talk to NPCs, complete quests, build structures, and explore with other players in real time.

Today it works. Here’s how I built the whole thing.

What Worldforge Actually Is

Worldforge is two things at once:

A worldbuilding platform — an entity library with six types (characters, locations, factions, artifacts, species, events), a visual timeline grouped by eras, a force-directed connections graph, a drawable world map with faction territories, and an AI storytelling engine that generates ongoing narratives for your characters.

A playable game — a full HTML5 Canvas 2D game engine running inside a React component. Three explorable zones, 25+ NPCs with branching dialogue, 8 quest chains, real-time melee combat, a 45-item building system, resource gathering, shops, enemy AI with state machines, and WebSocket multiplayer with PvP combat.

Every world you create on the platform becomes both. The lore tools let you design the story. The game lets you live inside it.

The Stack

  • Framework: Next.js 16 (App Router, React 19, Server Components)
  • Database: Prisma 6 + PostgreSQL (14 models, 8 enums)
  • Auth: Custom JWT with jose + bcryptjs (no third-party provider)
  • AI: Claude API via Anthropic SDK (storytelling), Google Gemini 2.0 Flash via OpenRouter (world generation)
  • Game Engine: HTML5 Canvas 2D, custom game loop, Simplex noise for terrain
  • Multiplayer: WebSocket server (ws library), custom protocol
  • Visualization: D3 force-directed graph, Canvas-based world map
  • Assets: ERW Grass Land 2.0, Summer Village pack, GuttyKreum NPC tilemap, Fantasy RPG sprites

The game engine alone — WorldExplore.tsx — is over 10,000 lines. It handles terrain rendering, sprite animation, NPC AI, combat physics, dialogue trees, building placement, quest state, ambient effects, and multiplayer synchronization, all in a single React component running at 60fps on a Canvas element.

World Generation: One Sentence to a Living Universe

The core pitch: describe your world in a sentence, AI builds it in 30 seconds.

When you click “Create World” and toggle AI mode, you type something like “a dying world where the last city floats above ash” and hit generate. The app sends your concept to Gemini 2.0 Flash, which returns structured JSON containing:

  • World metadata (title, tagline, description)
  • 2 historical eras with date ranges and descriptions
  • 4 entities of mixed types (characters, locations, factions, artifacts, species)
  • 3 timeline events with era references and entity links
  • 3 entity relationships (alliances, rivalries, origins)

All of this gets hydrated into the database in one transaction. The UI shows a step-by-step progress animation (“Imagining your world…”, “Creating characters & locations…”). After ~30 seconds, you have a fully populated world with a timeline, a connections graph, entities with lore, and a game you can walk into.

You can also skip AI entirely and create worlds manually. Both paths lead to the same toolset.

The Game Engine: 10,000 Lines in One Component

The explore mode is a full 2D pixel-art game running on a 960×640 Canvas. The architecture is simple but rigid:

  • One useEffect runs the game loop. Its dependency array is [ready, ga] and nothing else.
  • All game state lives in refs (useRef). The game loop reads .current on every frame.
  • React state is only used for things that need to trigger UI re-renders outside the canvas (health bar, gold counter, dialogue overlays).
  • Terrain is pre-rendered to offscreen canvases on zone load. Only sprites, NPCs, and the player are drawn per frame.

This pattern — refs for game state, React state for UI — is the only way to run a 60fps game loop inside React without the useEffect tearing down and restarting on every state change. I learned this the hard way on day 2.

Rendering Pipeline

Each frame:

  1. Clear the canvas
  2. Stamp the pre-rendered terrain offscreen canvas (scrolled to camera position)
  3. Stamp the pre-rendered decoration offscreen canvas
  4. Collect all dynamic sprites (player, NPCs, enemies, remote players, placed buildings, animals, ambient effects) into a drawable array
  5. Y-sort the array (things lower on screen draw on top of things higher up)
  6. Draw each sprite with appropriate animation frame, hue-shift, rotation
  7. Draw HUD, dialogue boxes, damage numbers, banners on top

The Y-sorting is critical for depth — a player standing behind a tree should be occluded by it, a player in front should draw on top. Every drawable has a Y-position, and the array is sorted each frame before rendering.

Three Zones, Three Experiences

The Hub (100×50 tiles)

Your home base. Procedurally generated terrain using Simplex noise — grass, cobblestone paths, water features, shore tiles. The hub has:

  • A town merchant with 3 dialogue choices (shop, news, info) and context-aware responses that change based on your progress
  • 8 ambient townsfolk from the GuttyKreum sprite pack, wandering with pathfinding and random speech bubbles
  • A campfire that heals +3 HP/s and a well that heals +5 HP/s when you stand near them
  • Dark ruins that drain your health if you walk into them
  • Two zone gates: Northern Pass (to Grassland) and Docks South Gate (to Village), each with marble pillars, fence wings, and a signpost
  • Walk-over gold coins scattered around for early income
  • A Shiba Inu that wanders around and barks

The hub is where you build. Press B and you get a tabbed build menu with 45 items across structures, props, decoration, and functional buildings. Place a cabin, rotate it with R, watch it snap to the grid with a placement gizmo showing corner brackets and a direction arrow. Placed structures persist to the database and block movement with rotation-aware collision footprints.

After placing enough structures, named NPCs start moving in. Helena the Tavern Keeper appears after 5 buildings and offers healing plus a firewood-gathering quest. Marcus the Archivist shows up next with a discovery quest. Eight total residents can move into your settlement, each with unique dialogue trees and quest chains.

The Grassland (80×60 tiles)

The combat zone. A hand-crafted tile map with edge dithering, per-tile texture variation, animated water, and ambient effects — birds flying overhead, ducks swimming in a pond, frogs hopping near water, butterflies, campfire smoke, animated lamps, chimney smoke.

The grassland has 7 orc warriors and 1 orc shaman, each running a full AI state machine:

  ↓ (player hits)                                           ↓ (cooldown)
hurt ←——————————————————————————
  ↓ (HP ≤ 0)

Each orc has 9 sprite sheets (walk, attack, hurt, death for two color variants plus the shaman). They chase you, take damage with floating red numbers, flinch on hit, and play a full death animation before dropping +10 gold.

The zone has 8 points of interest: the Orc Stronghold (fortified camp with tents, walls, flags), Vendor Camp (safe zone with a merchant selling potions and gear), Dark Cave, Rocky Overlook, Forest Clearing, Stream Crossing, Ruined Cart, and Frog Pond. Each has E-key inspect text and registers as a discovery.

The mission chain: visit 5 POIs, then kill all 7 orcs, then a reward chest spawns with +75 gold. There’s also a Combat Shrine — clear the orcs near it, press E to activate, get +50% damage and -20% damage taken for 45 seconds.

The Seaside Village (40×30 tiles)

The social zone. A coastal settlement with 6 named NPCs, each with 3 dialogue choices:

  • Fiona — food merchant (bread, fish, berries)
  • Gerald — general trader (tools, supplies)
  • Elder Rowan — village elder (lore, quest offers, history)
  • Marina — quest-giver for “Marina’s Lost Necklace”
  • Tom — shepherd (area info, rumors)
  • Nana Rose — cottage grandmother (stories, tea healing)

Plus Witch Willow, a fortune teller in a cottage at the village edge who sells magical potions (Barrier Hex for defense, Third Eye for awareness, Vigor Brew for strength) and tells random fortunes.

The village has 4 hostile bandits (using orc sprites but different behavior), 12 passive wildlife (ducks, frogs, butterflies), and the necklace quest — talk to Marina, search 3 locations (well, dock crates, garden flowers), find the necklace, return it for +30 gold plus a bonus chest.

The Combat System

Press SPACE to swing. Your character plays a 0.35-second attack animation with a visible slash arc in the direction you’re facing. Any enemy within 1.5 tiles in your attack cone takes damage.

Base attack: 15 damage. Strength buffs multiply by 1.5×. The Shrine buff stacks. The orc warriors have 60 HP each, the shaman has 80. With buffs you can three-shot a warrior.

Every hit spawns a floating damage number that rises and fades. Red for damage taken, gold for gold picked up. Enemies flash white on hit, play a hurt animation, then resume chasing.

Death costs 20% of your gold. You respawn at the zone’s entry point with full health, a “You have fallen…” banner, and 2 seconds of invincibility.

The Building System

Owners can press B to open a build menu with 45 items across 4 categories:

  • Structures: Fence, Fence Gate, Campfire, Barricade, Canvas Tent, Small Cabin, Watchtower, Stone Wall, Large Tent
  • Props: Barrel, Crate, Wood Logs, Spikes, Wooden Table, Weapon Rack, Water Well, Stone Bridge
  • Decoration: Sign Post, Pine Tree, Bush, Wildflowers, Mushrooms, Lamp Post, Boulder, Sunflowers, Dead Tree, Bone Trophy, Oak Tree, Stone Altar, Stone Gate, War Banner, Victory Banner, Dragon Fossil
  • Functional: Market Stall, Garden, Food Stall, Forge, Inn, Healing Shrine, Warehouse, Trade Wagon, Town Hall, and more

Three resources: Wood (start 30), Stone (start 15), and Gold. You gather wood and stone from forage spots with E, earn gold from combat and quests, and spend it all on buildings.

Placement validation checks: walkable tiles only (no water, no deep forest), no overlap with existing structures, not too close to zone gates. Items with rotation support swap their width and height dimensions when rotated 90°/270°, and the collision footprint adjusts accordingly.

The placement gizmo is satisfying — a raised 3D platform appears under the cursor with corner brackets, a direction arrow, and a scanning animation. Green for valid, red for blocked. Click to place, and the structure is immediately rendered with Y-sorted depth and sent to the database via API.

Functional buildings attract named NPC residents at milestone thresholds. Eight residents total, each with dialogue, quests, and personality.

28 Playable Characters

Press C during gameplay to open the character picker. 28 characters from the Fantasy RPG Sprite Pack — paladins, priests, dwarves, elves, warriors, mages, fire witches, herbalists, skeleton variants, and more.

Each character is drawn from a 352×400 sprite sheet with 4 directional rows (down, left, right, up) and 8 walk frames per direction. The picker shows a live preview with walk animation.

On top of character selection, there’s a hue-shift color wheel (0–360°) that recolors the entire sprite in real time. Pick a paladin, shift it cyan, and you’ve got a unique look. The hue shift persists across sessions and syncs to other players in multiplayer.

Real-Time Multiplayer

A WebSocket server (server/multiplayer.ts) runs on port 3001 alongside the Next.js app. When you enter a world, the client connects and joins a room keyed by the world’s slug.

The protocol is lightweight:

  • join — send your name, character index, hue shift, zone
  • move — throttled to 15 updates/sec, only sent on position change
  • attack — broadcasts attack animation to others
  • damage — PvP hit notification with target and amount
  • death — respawn notification
  • chat — 200-character messages

Remote players are interpolated on the client side — positions lerp smoothly to their target coordinates instead of snapping, which hides network latency. Each remote player renders as a GuttyKreum sprite with their chosen hue shift, a floating name label, and a health bar that appears when they take damage.

PvP Combat

Players can fight each other. When you press SPACE, the game checks for NPC hits first (existing behavior unchanged), then checks for remote players in your attack cone. Hit detection uses the same range and cone logic as NPC combat.

PvP damage follows the same rules: base damage, strength buffs, defense buffs, shrine buffs all apply. Damage numbers appear on the target. Death costs 20% gold with a “Slain by [PlayerName]!” banner. 2-second invincibility after respawn.

Remote player attacks render as red slash arcs at their position, so you can see when someone swings even if they’re not hitting you.

The Lore Layer

Every world has a full suite of worldbuilding tools layered on top of the game:

Entity Library

Six entity types: Character, Location, Faction, Artifact, Species, Event. Each has a title, summary, rich-text lore content, custom facts (key-value pairs like “Age: 34” or “Title: Exiled Prince”), freeform tags, an accent color, and positioning on both the world map and connections graph.

Entities have lifecycles — they can be “introduced” in one era and “retired” in another. The era filter on the connections graph and timeline respects this, showing how your world evolves over time.

Timeline

Events grouped by era, displayed chronologically. Each event has a date label, title, summary, impact assessment, and links to involved entities. Events can be manually created or generated by the AI storytelling system.

Connections Graph

A D3 force-directed graph where entities are nodes and relationships are edges. Nodes are color-coded by type. Edges carry labels like “allied with”, “betrayed”, “founded”. You can drag nodes to arrange the layout, and positions persist.

Relationships also have lifecycles — formed in one era, dissolved in another. The era filter lets you scrub through time and watch alliances form and break.

World Map

A canvas-based map where you place entity pins at coordinates, draw region polygons, and assign faction territories per era. The territory system creates a visual history of who controlled what and when.

AI Storytelling

This is the feature that ties the lore tools to ongoing narrative generation.

Pick any character entity and enable AI Storytelling. You configure three things:

  1. Personality prompt: How should the AI write this character? (“Cautious, intellectual, driven by duty but privately doubts the Council’s motives…”)
  2. Scope: Safe (journals, travel, conversations), Expanded (encounters, relationship shifts, minor conflicts), or Advanced (major events, faction shifts, bold twists)
  3. Constraints: What should the AI never do? (“Never kill named characters, stay within the floating citadels…”)

Click “Write Next Story” and the system calls Claude with the character’s personality, constraints, full world context (other entities, recent events, existing relationships), and generates a narrative development. This could be a journal entry, an encounter, a discovery, a conversation, a rumor, or a relationship shift.

Generated stories land in a review queue. Each one shows the full narrative text and any proposed changes to the world state — new relationships, updated facts, new tags. You can:

  • Approve — applies all changes, creates a timeline event, marks it as canon
  • Edit — modify the text before approving
  • Reject — discard it, no changes applied

This creates a loop: the AI generates stories that propose changes to the world, you curate them, and approved stories become the context for future generations. Your world grows through a collaboration between your vision and the AI’s creativity.

The NPC Dialogue System

Every NPC in the game has branching dialogue triggered by pressing E within range. The dialogue box renders a bordered panel with the NPC’s name, role, text, and up to 3 numbered choices.

Dialogue is context-aware. The hub merchant’s “News” option checks how many orcs you’ve killed, whether you’ve completed the grassland mission, how many buildings you’ve placed. “Travelers say the orc presence is getting worse” becomes “The pass is much safer now, thanks to you” after you clear the stronghold.

Building residents track your progress: “You’ve built 5 structures, impressive!” or “The settlement is really taking shape with 15 buildings.”

Quest-giving NPCs track quest state: accepted, in-progress, completed, reward collected. Marina won’t offer her necklace quest twice. Barton won’t congratulate you for killing orcs until you actually have.

Ambient Life

The world feels alive because of dozens of small ambient systems running simultaneously:

  • Townsfolk: 8 NPCs in the hub wander between waypoints with NPC-specific walk speeds and wait times, facing the direction they’re moving, with random speech bubbles every 10–18 seconds
  • Animals: Sheep that wander and bleat, a Shiba Inu that barks, ducks that swim in ponds, frogs that hop near water, butterflies that flutter in random patterns
  • Birds: Fly overhead in arcs across the grassland sky
  • Particles: Nature particles (leaves, pollen) drift across the scene, campfire smoke rises, chimney smoke puffs from the stronghold, flies buzz near dark areas
  • Animated props: Lamps flicker, flags wave at the stronghold, water tiles animate
  • Warrior guards: Two armored guards stand at the grassland entrance, cycling through idle animation frames, with ambient dialogue about the orc threat

None of these affect gameplay. They exist purely to make the world feel inhabited rather than static.

The Database Schema

14 Prisma models power the whole thing:

  • User — auth, profile
  • World — title, tagline, description, visibility, cover gradient
  • WorldMember — collaboration (owner/editor/viewer roles)
  • Entity — 6 types, rich content, facts, tags, map/graph positions, era lifecycle
  • EntityRelation — connections between entities with labels and era lifecycle
  • Era — time periods with sort order, date ranges, colors
  • Event — timeline events linked to eras and entities
  • EventEntityLink — many-to-many event↔entity relationships with roles
  • ActivityLog — full audit trail of every action
  • MapRegion — drawable geographic areas
  • TerritoryRecord — faction ownership per region per era
  • PlacedObject — user-built structures in the game (zone, position, item type, rotation)
  • CharacterSession — AI storytelling configuration per character
  • Development — AI-generated story episodes with review status

Authentication & Access

Custom JWT auth, no third-party providers. Register with email/username/password, login returns a signed JWT stored in an httpOnly cookie with 7-day expiry. Server components call getSession() which verifies the token and returns the user payload.

Worlds can be private (only the owner sees them) or public (visible on the Discover page). The Discover page shows a grid of public worlds with gradient-colored cards showing entity and event counts.

There’s a play mode at /play/[slug] that lets anyone try a demo world without an account. Non-owners get the character picker; world owners keep their default sprite.

Lessons From 10 Days of Building

The ref pattern is non-negotiable for Canvas games in React

The game loop useEffect depends only on [ready, ga]. Every piece of game state — player position, health, cooldowns, quest progress, NPC states — lives in useRef and is read via .current inside the loop. If you put any of these in the dependency array, the effect tears down and restarts, killing the game loop.

React state is only used for values that need to trigger DOM re-renders outside the canvas (HUD health bar, gold counter). The canvas itself is a black box that React never re-renders — the game loop handles all drawing.

Y-sorting makes or breaks 2D depth

Every frame, all visible sprites (player, NPCs, enemies, buildings, trees, animals) go into a single array, sorted by Y position. Things lower on screen draw on top of things higher up. Without this, a player behind a tree would render on top of it, and buildings would have no depth.

Buildings with rotation complicate this — a 3×2 structure rotated 90° becomes 2×3, and its Y-sort anchor needs to be the bottom edge of the rotated footprint, not the original.

Offscreen canvases for terrain are a massive win

Drawing 80×60 tiles (4,800 tiles) every frame at 60fps would be brutal. Instead, terrain and static decorations are pre-rendered to offscreen canvases once when the zone loads, then stamped onto the visible canvas each frame as a single drawImage call. The game loop only draws dynamic elements (sprites, animations, UI).

NPC dialogue needs more state than you think

A dialogue system that looks simple (“NPC says X, player picks 1/2/3”) actually requires tracking: is a dialogue currently open? Which NPC? Which set of choices? Has the player already completed this quest? What’s their kill count? Building count? Discovery count? Are they in a shop submenu? Did they just close the dialogue (debounce the E key)?

Each NPC ends up needing a small state machine of its own, on top of the dialogue content.

Multiplayer is simpler than it sounds

The entire server is under 300 lines. Join a room, broadcast moves, handle disconnects. PvP adds attack/damage/death messages. The client does all the heavy lifting — interpolation, hit detection, visual feedback. The server is a dumb relay.

The hard part isn’t the networking — it’s making sure the game loop reads multiplayer state from refs without adding dependencies that restart the loop.

What’s Next

Worldforge is live and playable. You can create a world with AI or from scratch, explore it as a game, build a settlement, fight enemies, complete quests, manage lore with timelines and graphs, have AI write ongoing stories for your characters, and play with others in real time.

The codebase is a monolith by design. One component handles the game. One server handles multiplayer. One database handles everything else. The simplicity is the point — it means a single developer can hold the entire system in their head and ship features in hours instead of weeks.

Ten days. One Next.js app. A worldbuilding platform and a full game engine, running in the same browser tab.
